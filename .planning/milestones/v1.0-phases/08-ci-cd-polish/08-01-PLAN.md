# Plan 08-01: CI/CD Polish

## Goal

Apply all 8 CI/CD polish items from deep-review findings H2, M7, M11, L2, L3, L4, L10, L12.
These are independent config and file changes with no source code impact.

## Context

All changes are in `.github/workflows/`, `package.json`, and `slack-app-manifest.yaml`.

**Source of truth for each fix:** `.planning/reviews/2026-03-27-deep-review.md`

## Tasks

<task id="08-01-T01" title="Fix CI push trigger — branches main only (M7)">

**File:** `.github/workflows/ci.yml`

Change the push trigger from `branches: ["*"]` to `branches: ["main"]`.

**Before:**
```yaml
on:
  push:
    branches: ["*"]
  pull_request:
```

**After:**
```yaml
on:
  push:
    branches: ["main"]
  pull_request:
```

**Why:** Push on `"*"` plus `pull_request` runs CI twice for every push to a PR branch — wastes CI minutes and creates duplicate status checks.

</task>

<task id="08-01-T02" title="Add security audit step to CI (L12)">

**File:** `.github/workflows/ci.yml`

Add a `bun audit` step after the test step. This provides basic npm dependency security scanning for open-source hygiene.

```yaml
      - run: bun audit
```

Note: `bun audit` is the Bun equivalent of `npm audit`. If `bun audit` is not available in the Bun version used by `oven-sh/setup-bun`, fall back to `npm audit --audit-level=moderate` (requires `actions/setup-node` to be present). Check Bun docs — as of Bun 1.2+ `bun audit` is supported.

**Placement:** After `bun test --coverage`, before any future steps.

</task>

<task id="08-01-T03" title="Pin GitHub Actions to specific releases (M11)">

**Files:** `.github/workflows/ci.yml`, `.github/workflows/release.yml`

Pin all `uses:` references from floating major tags to specific release versions. Use the latest stable releases with a comment showing the tag for human readability.

**Pins to apply:**
- `actions/checkout@v4` → `actions/checkout@v4.2.2`
- `oven-sh/setup-bun@v2` → `oven-sh/setup-bun@v2.0.3`
- `actions/setup-node@v4` → `actions/setup-node@v4.2.0`

**Format** (add tag comment for human readability):
```yaml
- uses: actions/checkout@v4.2.2       # v4.2.2
- uses: oven-sh/setup-bun@v2.0.3      # v2.0.3
- uses: actions/setup-node@v4.2.0     # v4.2.0
```

**Why:** Floating major tags (`@v4`, `@v2`) can silently pick up breaking changes. Pinning + Dependabot (T04) gives a controlled update path.

</task>

<task id="08-01-T04" title="Add .github/dependabot.yml (L2)">

**File:** `.github/dependabot.yml` (new file)

Create Dependabot configuration with entries for both `npm` and `github-actions` ecosystems.

```yaml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 5

  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 5
```

**Why:** No Dependabot config means no automated dependency updates for either npm packages or GitHub Actions versions. Weekly cadence is appropriate for a CLI tool.

</task>

<task id="08-01-T05" title="Add tag-version consistency check to release workflow (H2)">

**File:** `.github/workflows/release.yml`

Add a pre-publish validation step that exits with code 1 if the git tag does not match `package.json` version.

Add **before** the `bun install` step:

```yaml
      - name: Validate tag matches package.json version
        run: |
          PKG_VERSION=$(node -e "const p=require('./package.json'); console.log(p.version)")
          TAG_NAME="${{ github.ref_name }}"
          if [ "v${PKG_VERSION}" != "${TAG_NAME}" ]; then
            echo "ERROR: Tag ${TAG_NAME} does not match package.json version v${PKG_VERSION}"
            exit 1
          fi
          echo "Version check passed: ${TAG_NAME} matches package.json"
```

**Why (H2):** A tag typo (e.g., `v0.2.0` when `package.json` says `0.1.0`) publishes the wrong version to npm, creating a permanent mismatch between the git tag, GitHub Release, and npm registry. This is a high-severity finding.

</task>

<task id="08-01-T06" title="Add bun test --coverage to release workflow (L10)">

**File:** `.github/workflows/release.yml`

Change the test step from `bun test` to `bun test --coverage` to match CI behavior.

**Before:**
```yaml
      - run: bun test
```

**After:**
```yaml
      - run: bun test --coverage
```

**Why:** Consistency between CI and release workflow ensures the release gate is as strict as the CI gate.

</task>

<task id="08-01-T07" title="Add biome lint to prepublishOnly (L3)">

**File:** `package.json`

Update the `prepublishOnly` script to include `bunx biome check .` before the existing typecheck and test steps.

**Before:**
```json
"prepublishOnly": "bunx tsc --noEmit && bun test"
```

**After:**
```json
"prepublishOnly": "bunx biome check . && bunx tsc --noEmit && bun test"
```

**Why:** Local `npm publish` currently skips biome lint. This ensures any manual publish path enforces the same quality gates as CI.

</task>

<task id="08-01-T08" title="Add DM channel unsupported comment to slack-app-manifest.yaml (L4)">

**Placement:** Before or after the existing scope entries, clearly visible to operators configuring the manifest.

</task>
