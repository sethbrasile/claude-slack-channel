# Plan 06-01: Shutdown Guard + ts-Missing Logging

## Objective

Apply three targeted fixes to `src/server.ts` and `src/slack-client.ts` that close the shutdown idempotency gap (M1) and add diagnostic logging for events without a `ts` field (L1). Add corresponding unit tests.

## Context

**M1 — Shutdown idempotency:** Three signals can fire simultaneously (SIGTERM + SIGINT + stdin close). Without a guard, `shutdown()` runs multiple times, calling `socketMode.disconnect()` and `server.close()` in parallel. Fix: add a `shutdownInitiated` boolean guard. Also capture `messageQueue` into a local const immediately after `socketMode.disconnect()` resolves to prevent a post-disconnect message from extending the drain target.

**L1 — Missing-ts logging:** When `event.ts` is missing or empty, the current code silently discards the event (via `if (!ts || seenTs.has(ts)) return`). The success criteria requires logging `[slack-client] event without ts` to stderr before discarding. The seenTs guard (`!ts` check before `seenTs.set`) already prevents pollution — we just need the log line.

**stdout is sacred:** All logging MUST use `console.error()`, never `console.log()`.

<title>Add shutdownInitiated guard to server.ts</title>
<file>src/server.ts</file>
<action>
In the `if (import.meta.main)` block, add a `let shutdownInitiated = false` variable.

Modify `shutdown()` to check and set the guard:

```typescript
let shutdownInitiated = false

async function shutdown(signal: string): Promise<void> {
  if (shutdownInitiated) {
    console.error(`[shutdown] already in progress, ignoring ${signal}`)
    return
  }
  shutdownInitiated = true
  console.error(`[shutdown] ${signal}`)
  try {
    await socketMode.disconnect()
  } catch (err) {
    console.error('[shutdown] socketMode.disconnect failed:', safeErrorMessage(err))
  }
  const drainQueue = messageQueue
  try {
    await drainQueue
  } catch (err) {
    console.error('[shutdown] messageQueue drain failed:', safeErrorMessage(err))
  }
  try {
    await server.close()
  } catch (err) {
    console.error('[shutdown] server.close failed:', safeErrorMessage(err))
  }
  process.exit(0)
}
```

Key changes:
1. `if (shutdownInitiated) { ... return }` guard — second call is a no-op
2. `shutdownInitiated = true` set before any async ops
3. `const drainQueue = messageQueue` — capture reference after disconnect resolves
4. Await `drainQueue` instead of `messageQueue`
</action>

Since the ts guard behavior cannot be unit-tested without mocking `SocketModeClient`, add a descriptive test that documents the expected behavior and tests the guard logic pattern through a simple extracted validation:

Add a new `describe` block for ts-guard behavior documentation. If direct testing is not feasible without mocking the SDK, add a comment explaining that the ts-guard behavior is verified by code review + the existing empty-ts behavior in the dedup logic.

**Preferred approach:** Extract the ts-check into a testable function. Add to `slack-client.ts`:

```typescript
/**
 * Returns the ts string if valid (non-empty), or null if missing/empty.
 * Logs a warning to stderr when ts is absent.
 * @internal exported for testing only
 */
export function validateEventTs(ts: string | undefined): string | null {
  const normalized = ts ?? ''
  if (!normalized) {
    console.error('[slack-client] event without ts')
    return null
  }
  return normalized
}
```

Then in the event handler:
```typescript
const ts = validateEventTs(event.ts)
if (ts === null || seenTs.has(ts)) return
if (ts !== null) seenTs.set(ts, now + DEDUP_TTL_MS)
```

**Simplification:** If extracting `validateEventTs` is too invasive, keep the inline change from task 06-01-02 and add tests that verify `console.error` is called with `'[slack-client] event without ts'` by directly testing the pattern. Document that the behavior is enforced by the inline code.

**Decision (pick one approach and be consistent):**
- Option A (extract `validateEventTs`): Cleaner, more testable
- Option B (inline + console.error spy): Simpler, fewer exports

For this plan, **use Option A** — extract `validateEventTs` and test it:

Add to `slack-client.test.ts`:
```typescript
import { ..., validateEventTs } from '../slack-client.ts'

describe('validateEventTs', () => {
  it('returns the ts string when present and non-empty', () => {
    expect(validateEventTs('1234567890.123456')).toBe('1234567890.123456')
  })

  it('returns null and logs when ts is undefined', () => {
    const spy = spyOn(console, 'error').mockImplementation(() => {})
    const result = validateEventTs(undefined)
    expect(result).toBeNull()
    expect(spy).toHaveBeenCalledWith('[slack-client] event without ts')
    spy.mockRestore()
  })

  it('returns null and logs when ts is empty string', () => {
    const spy = spyOn(console, 'error').mockImplementation(() => {})
    const result = validateEventTs('')
    expect(result).toBeNull()
    expect(spy).toHaveBeenCalledWith('[slack-client] event without ts')
    spy.mockRestore()
  })

  it('does NOT call onMessage for missing ts (no seenTs pollution)', () => {
    // Verified by: validateEventTs returns null -> handler returns early -> seenTs.set never called
    // This is a structural guarantee from the code, not a runtime test.
    // The above tests confirm null is returned, which causes the early return.
    expect(validateEventTs(undefined)).toBeNull()
    expect(validateEventTs('')).toBeNull()
  })
})
```

## Notes

- The `validateEventTs` export is marked `@internal` — it's a testing seam, not a public API
- If the extraction approach causes any issues (type errors, lint violations), fall back to Option B (inline change + a comment noting that seenTs pollution is prevented by the `if (!ts)` guard before `seenTs.set`)
- The `shutdownInitiated` variable must be declared with `let` (not `const`) since it's reassigned
- `shutdownInitiated = true` must be set BEFORE any `await` to prevent a race between the guard check and the first async operation
